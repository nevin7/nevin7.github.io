The Problem

Financial AI needs domain knowledge it can actually trust.

AI agents operating in banking — whether surfacing credit risk, flagging AML anomalies, or answering customer queries — routinely hallucinate because the underlying knowledge layer is informal or absent. Terms like "non-performing loan," "beneficial owner," or "Tier 1 capital" mean specific, legally significant things. A model that doesn't know that is a liability.

This portfolio demonstrates the full semantic stack required to ground a financial AI system: a governed SKOS taxonomy, a formal OWL ontology, and an instantiated Neo4j knowledge graph — all aligned to FIBO (the Financial Industry Business Ontology) and designed to plug directly into a RAG pipeline or agentic architecture.

What This Portfolio Demonstrates

  • FIBO alignment — terms mapped to the Financial Industry Business Ontology for interoperability with industry standards

  • AI-ready metadata — canonical definitions, usage constraints, and scope notes formatted for LLM consumption

  • Governance design — steward roles, approval workflows, and lifecycle states built into the taxonomy itself

  • Full OWL formalism — restrictions, inverse properties, and SKOS ConceptScheme bridging, openable in Protégé

  • Queryable knowledge graph — 6 synthetic customers, AML/SAR scenarios, fraud events, and 7 analytical Cypher queries

A formal OWL ontology in Turtle syntax that provides machine-readable, logically consistent definitions for the banking domain. The ontology declares classes with subclass hierarchies, object properties with domain/range constraints, OWL restrictions (cardinality, someValuesFrom, allValuesFrom), and a bridging SKOS ConceptScheme that connects the formal ontology to the taxonomy layer.

FIBO namespaces are imported throughout, ensuring the model is interoperable with the industry-standard financial ontology. Openable and reasoned in Protégé — inference-clean, no unsatisfiable classes.

banking_domain.ttl 
# Prefixes
@prefix bk:    <https://nevin-kohler.com/ontology/banking#> .
@prefix fibo-be: <https://spec.edmcouncil.org/fibo/ontology/BE/> .
@prefix skos:  <http://www.w3.org/2004/02/skos/core#> .
@prefix owl:   <http://www.w3.org/2002/07/owl#> .

# Class declaration with FIBO alignment
bk:NonPerformingLoan
  a owl:Class ;
  rdfs:subClassOf bk:Loan ,
    [ a owl:Restriction ;
      owl:onProperty bk:hasDelinquencyStatus ;
      owl:someValuesFrom bk:DelinquencyStatus ] ;
  skos:definition "A loan where the borrower has failed to
    make scheduled payments for 90+ days,
    triggering credit risk classification." ;
  rdfs:isDefinedBy fibo-be:LoanProducts .

# Object property with inverse
bk:hasCollateral
  a owl:ObjectProperty ;
  owl:inverseOf bk:collateralizesLoan ;
  rdfs:domain bk:SecuredLoan ;
  rdfs:range  bk:CollateralAsset .

7 Analytical Queries — Click to Expand

1. Non-Performing Loan Analysis with Collateral + expand 
// Find all NPL customers, their loans, collateral, and LTV
MATCH (c:Customer)-[:HAS_ACCOUNT]->(a:Account)
      -[:HAS_LOAN]->(l:Loan {status: 'NonPerforming'})
OPTIONAL MATCH (l)-[:SECURED_BY]->(col:Collateral)
RETURN c.name, l.loanId, l.balance, l.daysDelinquent,
       col.type, col.appraisedValue, l.ltvRatio
ORDER BY l.daysDelinquent DESC
2. Full AML / PEP Chain — Alerts to SAR + expand 
// Trace PEP → accounts → transactions → AML alerts → SAR
MATCH (c:Customer {isPEP: true})
      -[:HAS_ACCOUNT]->(a:Account)
      -[:HAS_TRANSACTION]->(t:Transaction)
      -[:TRIGGERED]->(alert:AMLAlert)
OPTIONAL MATCH (alert)-[:ESCALATED_TO]->(sar:SARFiling)
RETURN c.name, c.pepStatus, t.amount, t.country,
       alert.alertType, alert.riskScore, sar.filingDate
3. Multi-Risk Customer Flagging + expand 
// Find customers with 2+ risk flags across any risk type
MATCH (c:Customer)
WITH c,
  CASE WHEN c.isPEP THEN 1 ELSE 0 END +
  CASE WHEN (c)-[:HAS_ACCOUNT]->()-[:HAS_LOAN]->
    (:Loan {status:'NonPerforming'}) THEN 1 ELSE 0 END +
  CASE WHEN (c)-[*]->(:AMLAlert) THEN 1 ELSE 0 END
  AS riskScore
WHERE riskScore >= 2
RETURN c.name, c.segment, riskScore
ORDER BY riskScore DESC
4. Beneficial Ownership Graph (Commercial) + expand 
// Resolve beneficial ownership chain for commercial entities
MATCH path = (owner:Customer)
             -[r:OWNS*1..3]->
             (entity:CommercialEntity)
RETURN owner.name,
       [n IN nodes(path) | n.name] AS ownershipChain,
       [rel IN relationships(path) | rel.ownershipPct]
         AS ownershipPercentages

Key Takeaways

What this demonstrates for an AI or data team.

End-to-end semantic stack

Most practitioners can do one layer well. This portfolio shows taxonomy, formal ontology, and knowledge graph working together as a coherent system — which is what a production AI knowledge layer actually requires.

FIBO-aligned financial domain depth

FIBO alignment isn't checkbox work — it requires understanding the ontology's namespace structure, import patterns, and how its classes map to real banking concepts. The portfolio demonstrates genuine domain fluency, not surface familiarity.

Governance built in from the start

Steward roles, lifecycle states, approval workflows, and audit fields are embedded in the taxonomy model — not afterthoughts. That's the difference between a semantic artifact and a governable enterprise asset.

AI-ready, not just correct

Every term has a structured ai_canonical_definition, usage constraints, and scope notes specifically formatted for LLM grounding and RAG retrieval — not just human readability.
